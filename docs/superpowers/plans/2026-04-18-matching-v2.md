# Matching v2 — Self-Improving Match Engine

**Date:** 2026-04-18
**Branch:** `feature/matching-v2` (worktree `.worktrees/matching-v2`), stacked on `feature/workbench-v5`
**Author:** Jace / Opus 4.7 session
**Status:** Phase 0 in progress
**Scope:** Replace discrete tier-based matching with a continuous, explainable score that learns from every human override in the Workbench. "Industry leading" through feedback latency + explainability + cross-rater learning.

---

## Why

Today's matcher is a 3-strategy cascade returning discrete tiers (tier1 / tier2 / uncertain). Jessica, Steph, Scout, Keli, and Katie generate dozens of labeled decisions per day via the Resolve modal + advanceStage transitions — decisions that contain rich signal about which match rules are right and which drift. That signal is currently filed as ops logs (statusHistory jsonb) and never looped back into the matcher.

Flipping that signal into training data turns the Workbench into a self-tuning system. Every override Jessica makes today improves tomorrow's queue.

## Design principles

1. **No new infrastructure.** Postgres + pgvector (already in the DB playbook) + existing jsonb columns. No new service, no message queue, no vector DB.
2. **Score is explainable.** Every assignment carries a 0.0-1.0 score AND a per-feature breakdown. Jessica sees "0.62: BOL last-4 matched, weight within 4%, driver fuzzy 0.71" not just "tier2".
3. **Decisions are first-class.** Every human action on an assignment (confirm / flag_back / route_uncertain) writes a `match_decisions` row with features snapshot + score_before + action_taken. That's labeled data for training.
4. **Feedback latency ≤ 1 day.** Weight tuner runs nightly; changes propagate on next reconciliation pass. Near-real-time is Phase 5+ scope.
5. **Backward compatible.** Discrete tiers remain derivable from score (score ≥ 0.8 = tier1, etc.) until Phase 4 deprecates the tier enum.

## Phase breakdown

### Phase 0 — Scorer as pure function

**File:** `backend/src/plugins/dispatch/services/match-scorer.service.ts`

**Exports:**
- `interface MatchFeatures` — all signals the scorer considers (bolMatch, weightDeltaPct, hasPhoto, driverSimilarity, autoMapTier, wellAssigned, hasTicket, hasRate, deliveredOn)
- `interface MatchScore` — { score: number, drivers: FeatureContribution[], tier: "high" | "medium" | "low" | "uncertain" }
- `interface FeatureContribution` — { feature: string, value: number, weight: number, contribution: number }
- `interface ScorerConfig` — the weight vector (defaults live inline in Phase 0, externalized to DB in Phase 3)
- `function scoreMatch(features: MatchFeatures, config?: ScorerConfig): MatchScore`
- `const DEFAULT_CONFIG: ScorerConfig`

**Tests (`backend/tests/dispatch/match-scorer.service.test.ts`):**
- Perfect match (all features green) → score ≥ 0.95
- Only BOL match (everything else null/low) → score ~0.3-0.4
- No photo but everything else green → score drops by photo weight
- Driver fuzzy match at 0.5 → contributes proportionally
- Deterministic: same inputs produce identical score + drivers array
- Drivers sum to ~= score (explainability invariant)
- Unknown features in config are ignored (forward compat)

**Ships:** a pure function. No DB. No migration. No UI change. Mergeable independently.

### Phase 2 — match_decisions table + decision writer

**Migration:** `0012_match_decisions.sql`
```sql
CREATE TABLE match_decisions (
  id bigserial PRIMARY KEY,
  assignment_id int NOT NULL REFERENCES assignments(id),
  load_id int NOT NULL REFERENCES loads(id),
  user_id int NOT NULL REFERENCES users(id),
  action text NOT NULL,           -- "confirm" | "route_uncertain:needs_rate" | etc.
  features_snapshot jsonb NOT NULL,
  score_before numeric(4,3) NOT NULL,
  score_after numeric(4,3),
  tier_before text NOT NULL,
  tier_after text,
  notes text,
  created_at timestamptz NOT NULL DEFAULT now()
);
CREATE INDEX ON match_decisions(created_at);
CREATE INDEX ON match_decisions(action);
```

**Files:**
- `backend/src/plugins/dispatch/services/match-decisions.service.ts` (new) — `recordDecision(db, ctx)` called from advanceStage / routeUncertain / flagToUncertain
- `backend/src/plugins/dispatch/services/workbench.service.ts` (modify) — call recordDecision in each transition
- Tests: verify decisions row written with correct features snapshot for each transition path

**Ships:** labeled data accumulates from day 1. No tuner yet — just capture.

### Phase 3 — scorer_config table + nightly tuner

**Migration:** `0013_scorer_config.sql`
```sql
CREATE TABLE scorer_config (
  version int PRIMARY KEY,
  weights jsonb NOT NULL,
  trained_at timestamptz NOT NULL DEFAULT now(),
  training_rows int NOT NULL,
  metrics jsonb,                  -- precision/recall per action
  is_active boolean NOT NULL DEFAULT false,
  notes text
);
CREATE UNIQUE INDEX scorer_config_active_one ON scorer_config(is_active) WHERE is_active;
-- Seed row 1 = current defaults, active
```

**Files:**
- `backend/src/plugins/dispatch/services/scorer-tuner.service.ts` (new)
  - `trainWeights(decisions: MatchDecision[]): ScorerConfig` — pure function, logistic regression or simple gradient descent over features → confirm vs. route_uncertain outcomes
  - `tuneOnce(db, opts)` — reads decisions, trains, writes new config version with is_active=false for review
  - `activateConfig(db, version)` — flips is_active, matcher reloads
- `backend/src/scheduler.ts` (modify) — nightly job
- Cache invalidation: in-memory config reload when version changes (low-tech pub/sub via LISTEN/NOTIFY or polling)

**Ships:** self-tuning. But tuner output is REVIEW-gated — config lands inactive, human flips it after sanity check (Phase 3.5: auto-activate after N days passing guardrails).

### Phase 4 — matcher reads score, deprecates tiers

**Files:**
- `backend/src/plugins/dispatch/services/auto-mapper.service.ts` (modify) — cascade still runs for feature extraction, but routing decision (uncertain vs ready_to_build) uses score threshold + active scorer_config
- `backend/src/db/schema.ts` (modify) — `auto_map_tier` column becomes legacy (keep writing for now, read path switches to match_score)
- `frontend/src/pages/Workbench.tsx` (modify) — "Uncertain" filter becomes "Score < 0.6" or similar threshold-driven

**Tests:** full regression — the existing 141 tests must still pass. Add new ones proving tier buckets map correctly from score ranges.

**Ships:** the matcher actually uses the triad. Industry-leading behaviors (explainability + <24h feedback latency + 5-rater learning) become live.

## What's NOT in scope

- Embedding-based fuzzy match (#1 in the bigger discussion) — requires Phase 4 complete + pgvector setup + embedder choice. Future phase.
- Photo-as-source-of-truth (#2) — robust multi-field OCR is its own big project. Future phase.
- Pre-ingest anomaly detection (#4) — orthogonal, parallel track, separate branch.
- Real-time feedback (<1s latency) — Phase 5+.
- Cross-rater disagreement analysis (when Jessica + Steph label the same load differently) — Phase 5+.

## Success metrics (for Phase 4 cutover)

- 141 existing tests still green
- Median score drift per decision ≤ 0.05 (i.e., no thrashing on each new training row)
- Jessica's uncertain queue shrinks ≥ 15% vs current baseline within 2 weeks of Phase 3 active
- At least one weight tuned down from baseline by ≥ 20% within first tuning run (proves the loop is doing something)
